# Act on Ground - 事上练

## Overview

方向明确了就动手。不准备、不等条件、不找完美方案——找最小的一步，做了，让现实告诉你答案。每一步都是在真实事务中检验认知。

> **"知而不行只是未知。"** ——王阳明
> 你说你懂了但没做，那就是还没懂。真正的理解只有在行动中才能完成。
> **"知之真切笃实处即是行，行之明觉精察处即是知。"**
> 认识到位了自然就是行动，行动中有觉知自然就是认识。

## The Core Pattern

**WITHOUT this skill**:
```
清晰目标 → 列完整计划 → 准备工具 → 等条件成熟 → 发现准备不够 → 继续准备 → 循环
```

**WITH this skill**:
```
清晰目标 → 剥到最小一步 → 立刻做 → 看结果 → 知道了什么 → 下一步
```

## Trigger Conditions

- 方向已经清晰（通过 clarify-intent 或用户自身已想清楚）
- 但迟迟不动手："让我先看看""再研究研究""准备好了再说"
- 在多个方案间反复比较，超过 2 轮还没动手
- 说了"开始"但实际在做准备工作而非推进目标
- 用户说"我不知道从哪开始"——方向虽清但第一步被目标体量吓住了

## The Act-on-Ground Algorithm

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. 剥到最小 (STRIP) — 找最小可验证动作                           │
│    目标是 X。不做 X。                                            │
│    做能验证 X 是否走对的第一小块。                                 │
│    标准：                                                        │
│    · 1 小时内能完成                                              │
│    · 做完有一个明确的、可观测的结果                               │
│    · 结果能告诉你"继续"还是"调整"                                │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│ 2. 即刻行动 (ACT) — 做这一步                                     │
│    不准备更多。                                                  │
│    不优化做法。                                                  │
│    用最粗糙、最直接的方式完成这一步。                              │
│    "先跑起来再说"不是偷懒，是策略。                               │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│ 3. 事上验证 (VERIFY) — 检查结果                                  │
│    这一步产出了什么？                                            │
│    产出是否可观测？（能说出来、能看见、能量化）                    │
│    ┌─ 有观测结果 → 进入学习                                      │
│    └─ 没有观测结果 → 这一步太空了，重新剥                        │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│ 4. 知行互证 (LEARN) — 行动揭示了什么                              │
│    做完这一步，你对目标的理解变了没有？                            │
│    · 变深了 → 继续下一步                                        │
│    · 发现前提错了 → 回到 clarify-intent 重新定向                  │
│    · 发现比想的简单 → 加大步伐                                   │
│    · 发现比想的复杂 → 缩小范围，继续小步                         │
└─────────────────────────────────────────────────────────────────┘
```

## Validation Gate

**过关心**：这一步有没有产生**可观测的结果**？

| 结果类型 | 是否过关 | 示例 |
|---------|---------|------|
| 代码运行通过了 | ✅ 过关 | 跑通了 auth 流程 |
| 找到了具体数字 | ✅ 过关 | 5 个 API 中 2 个超过 3 秒 |
| 能展示/演示的东西 | ✅ 过关 | 画出了调用关系图 |
| "了解了大致情况" | ❌ 不过关 | 了解了什么？说不出来就是没结果 |
| "看了一下代码" | ❌ 不过关 | 看到了什么？没有产出物 |
| "有了个思路" | ❌ 不过关 | 思路不是结果，做了才是 |

**核心判断**：如果有人问"你做了什么"，你能否用**事实**而非**感受**回答？不能 = 这一步在空转。

## Example Output

```markdown
# Act on Ground: "找到认证相关文件，画调用关系图"

## Step 1: 剥到最小
目标：认证系统重构。第一步不必是重构。
最小可验证动作：grep 出所有引用 auth 的文件，数一下有多少个。

## Step 2: 即刻行动
```
grep -rl "auth" --include="*.ts" ./src
```

## Step 3: 事上验证
结果：17 个文件引用了 auth。
其中 6 个直接 import auth module，11 个间接依赖。

可观测：✅ 有具体数字，有文件列表。

## Step 4: 知行互证
原来以为影响范围是 7 个文件（用户说的）。
实际是 17 个。复杂度比预期高。

行动揭示的认知变化：
- 比用户感知的范围大了 2.4 倍
- 但其中 11 个是间接依赖，可能不需要改
- 下一步应该验证：这 11 个间接依赖，改 auth 时真的会断吗？
```

## More Examples

| 目标 | 最小一步 | 观测结果 | 认知变化 |
|------|---------|---------|---------|
| dashboard 首屏 < 2s | 给 5 个 API 调用加耗时日志 | 2 个 API 各 > 3s | 问题集中在这 2 个，不是全面性能问题 |
| 功能开发周期 < 1 天 | 找上次加功能改的文件列表 | 改了 7 个文件，其中 3 个是重复逻辑 | 有重复代码可以抽公共模块 |
| 搜索准确率提升 | 收集 10 个搜不准的 case | 7/10 是关键词匹配问题，不是排序问题 | 不需要换搜索引擎，需要改分词策略 |
| 可用性 > 99.9% | 分析最近 3 次宕机日志 | 都是同一个服务的 OOM | 单点问题，不需要全面加固 |
| 改 bug 不引出新 bug | 看最近 3 次回归的 diff | 3 次都动了同一个共享模块 | 这个模块缺少隔离，改一个影响一片 |

## 行动粒度参考卡

目标大的时候容易卡住，因为你看到的是整座山。以下是剥到最小的参考：

| 你看到的 | 你以为要做的 | 最小一步 | 花多久 |
|---------|------------|---------|-------|
| 重写一个模块 | 读完所有代码再设计 | 找到这个模块被谁调用，数一下调用点 | 30 分钟 |
| 解决性能问题 | 全链路压测 | 在最慢的地方加一行计时日志 | 10 分钟 |
| 修复一个 bug | 理解整个系统 | 复现这个 bug，记录复现步骤 | 20 分钟 |
| 加一个新功能 | 设计完整方案 | 写一个最粗糙的 PoC，只覆盖主路径 | 1 小时 |
| 技术选型 | 调研所有方案 | 用第一个方案跑一个 hello world | 30 分钟 |

**关键原则**：最小一步不是"计划的第一步"，而是"能最快告诉你走对了还是错了的一步"。

## 准备陷阱识别

以下是"假装在行动"的常见模式：

| 陷阱 | 表现 | 破局方法 |
|------|------|---------|
| **调研成瘾** | "让我先看看别人怎么做的" | 看第一个就行，看完立刻动手 |
| **工具准备** | "先搭个环境/配个工具" | 用手头最简陋的方式先做 |
| **完美起步** | "要设计好再动手" | 先用最丑的方式做出来 |
| **信息焦虑** | "再查一个资料就开始" | 查到第三个就停，动手 |
| **范围蔓延** | "既然要做不如把 XX 也一起做了" | 只做当前这一步 |

> **"知是行之始，行是知之成。"**
> 知是行动的开始，行动是知的完成。停在"准备"阶段，就是有始无终——说明那个"知"还不够真。

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| 把"最小一步"定得太小，做完了什么都没学到 | 最小一步必须能验证一个假设，纯操作不算 |
| 把"最小一步"定得太大，做着做着又卡住 | 如果超过 1 小时还没结果，再剥一层 |
| 跳过验证直接进下一步 | 每一步必须过验证关，否则是在盲走 |
| 事后合理化——结果不好也强行说"学到了" | 说清楚具体学到了什么。说不清 = 没学到 |
| 做完一步又回到规划模式 | 做完一步立刻问：下一步最小动作是什么？不要回到白板 |

## Combined With

- **clarify-intent**: 事上练发现方向有问题时，回到明心定向
- **reflect-anchor**: 连续做了几步之后，进入复盘校准
- **constraint-flip**: 行动中遇到阻碍时，翻转约束
- **cross-domain**: 卡在某一步不知道怎么做时，用跨域类比找灵感
