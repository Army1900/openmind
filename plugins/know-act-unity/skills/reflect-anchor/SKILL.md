# Reflect Anchor - 知行复盘

## Overview

做了一段时间，停下来照镜子。方向偏了没有？行动验证了什么？推翻了什么？"我以为对但其实错的是什么？"——答不上来这道题，复盘就是走过场。

> **"知之真切笃实处即是行，行之明觉精察处即是知。"** ——王阳明
> 认识真切到位了就是行动，行动中保持清醒觉察就是认识。
> 复盘不是事后总结，是让行动中积累的隐含认知浮出水面，变成明天的方向。

## The Core Pattern

**WITHOUT this skill**:
```
做了几步 → 感觉还行 → 继续做 → 做了很多 → 回头一看方向偏了 / 做了一堆没用的
```

**WITH this skill**:
```
做了几步 → 停下来 → 锚定原目标 → 检验行动vs预期 → 面对偏差 → 校准方向
```

## Trigger Conditions

- 完成了一个阶段（3-5 个 action step 之后）
- 一步失败了——预期的结果没出来
- 用户说"感觉不对""是不是跑偏了"
- 连续做了很多但没看过目标
- 用户说"做完了"但没对照目标检查

## The Reflect-Anchor Algorithm

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. 锚定初心 (ANCHOR) — 目标是什么？                               │
│    回到最开始的那个清晰方向（clarify-intent 的产出）               │
│    原话写出来，不改。                                             │
│    然后：现在做的事，和这个目标什么关系？                          │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│ 2. 事实验证 (EXAMINE) — 行动说了什么？                            │
│    列出做了的每一步和实际结果                                     │
│    哪些结果验证了原来的假设？                                     │
│    哪些结果推翻了原来的假设？                                     │
│    只看事实，不看期望。                                           │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│ 3. 直面未知 (CONFRONT) — 我以为我知道但其实不知道的是什么？        │
│    这是复盘的核心。                                              │
│    不是"哪里做错了"——那是追责思维。                              │
│    是"我带着什么错误假设在行动"——这是认知更新。                   │
│    ┌─ 能说出一两个 → 复盘有效                                   │
│    └─ 说不出来 → 要么复盘太浅，要么行动太安全没触碰到未知         │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│ 4. 校准方向 (CALIBRATE) — 更新认知，确定下一步                    │
│    基于更新后的认知：                                            │
│    · 目标还是原来的吗？                                          │
│      ┌─ 是 → 继续，但调整路径                                    │
│      └─ 不是 → 回到 clarify-intent                               │
│    · 下一步是什么？用 act-on-ground 剥到最小                     │
└─────────────────────────────────────────────────────────────────┘
```

## Validation Gate

**过关心**：能不能说出"**我以为对但其实错的**"是什么？

| 状态 | 含义 | 行动 |
|------|------|------|
| 能说出来，且具体 | 复盘有效，认知已更新 | 进入校准，确定下一步 |
| 能说出来，但很笼统 | 方向对了但不够深 | 再挖一层："这个错误假设是怎么形成的？" |
| 说不出来 | 两种可能：①复盘太浅 ②行动太安全 | ①重新审视每一步的结果 ②下一步要做点有风险的 |

**判断标准**：
- "我发现 auth 模块的影响范围比预期大 2 倍" → ✅ 具体的错误假设
- "我低估了复杂度" → ⚠️ 太笼统，继续挖：具体是哪个方面的复杂度？
- "一切都在按计划进行" → ❌ 如果没有认知变化，要么复盘走过场，要么行动太保守

## Example Output

```markdown
# Reflect Anchor: 认证系统重构 中期复盘

## Step 1: 锚定初心
原始目标："加认证相关功能时，改动集中在 1-2 个文件，1 天内完成。"

当前在做什么：正在梳理 auth 模块的依赖关系。

关系：直接相关。梳理依赖是为了找到改动的集中点。

## Step 2: 事实验证

| 步骤 | 预期 | 实际 | 结论 |
|------|------|------|------|
| grep auth 引用 | ~7 个文件 | 17 个文件 | 影响范围被低估 |
| 分析间接依赖 | 可能有少量 | 11 个间接依赖 | 耦合比想象的深 |
| 检查间接依赖是否真的会断 | 大部分安全 | 其中 8 个引用了内部实现 | 封装有问题，不是简单的重构能解决 |

## Step 3: 直面未知

**我以为对但其实错的**：
1. "重构认证系统"是个重构问题 → 错。这是个架构问题。17 个文件引用 + 8 个穿透封装 = 不是改改代码能解决的
2. "集中改动到 1-2 个文件"这个目标本身可能需要先做一层抽象 → 原来以为直接重构就行，现在发现需要先建抽象层

**这些错误假设是怎么形成的**：
- 只听了用户说"上次改了 7 个文件"，没追问是哪 7 个、为什么改这么多
- 把症状（改的文件多）当了病因（封装边界不清晰）

## Step 4: 校准方向

目标是否变化：核心目标没变（加功能 1 天完成），但路径要调整。
不能直接重构，需要先建一层抽象，把 auth 的内部实现和外部接口分开。

**下一步（act-on-ground）**：
最小一步：选 1 个间接依赖最重的调用点，尝试抽出接口，看改动量。
```

## More Examples

| 场景 | 锚定发现 | 面对的未知 | 校准方向 |
|------|---------|-----------|---------|
| 性能优化做了 3 步 | 目标是 < 2s，已从 8s 降到 4s | 以为瓶颈全在后端，但前端渲染也吃 1s | 目标不变，后端继续优化 + 前端也加入范围 |
| 连续修了 3 个 bug | 每个 bug 修复花了半天 | 以为是个别问题，但 3 个 bug 都出在同一个模块 | 暂停修 bug，先重构这个脆弱模块 |
| 新功能做了 demo | demo 能跑但很粗糙 | 以为技术方案已经确定，但 demo 暴露了数据模型的缺陷 | 回到 clarify-intent，重新思考数据模型 |
| 3 个方案都试了一遍 | 都不可行 | 以为问题是"选哪个方案"，其实是前提条件不满足 | 回到 clarify-intent，重新定义问题边界 |

## 复盘深度参考卡

### 复盘的三层深度

| 层级 | 问什么 | 典型产出 |
|------|-------|---------|
| **L1: 事实层** | 做了什么？结果是什么？ | 事件时间线 + 结果列表 |
| **L2: 认知层** | 哪些假设被推翻了？ | "我以为 X，其实是 Y" |
| **L3: 模式层** | 这个错误假设是怎么形成的？ | "我倾向于低估 XX""我习惯先动手再想" |

**必须达到 L2 才算过关。** L3 不强求但强烈建议——L3 发现的是你反复犯的系统性错误。

### 典型的"我以为对但其实错的"模式

| 错误假设 | 真相 | 出现频率 |
|---------|------|---------|
| "改一下就行" | 牵一发而动全身 | ⭐⭐⭐⭐⭐ |
| "这个应该很快" | 隐藏依赖让工作量翻倍 | ⭐⭐⭐⭐⭐ |
| "先做后面再优化" | 技术债越积越快 | ⭐⭐⭐⭐ |
| "用户需要这个功能" | 用户需要的是另一个东西 | ⭐⭐⭐⭐ |
| "这个方案很成熟" | 在我们的场景下水土不服 | ⭐⭐⭐ |
| "做完就完了" | 维护成本是开发的 3 倍 | ⭐⭐⭐ |

## 失败复盘 vs 阶段复盘

两种复盘的侧重点不同：

| 维度 | 失败复盘 | 阶段复盘 |
|------|---------|---------|
| 触发 | 一步做砸了 | 做了几步，暂停看看 |
| 情绪 | 可能有挫败感 | 相对平静 |
| 重点 | 这个失败暴露了我什么盲区 | 这几步走下来方向对吗 |
| 核心问题 | "什么假设导致我走向了这个失败？" | "我的认知地图更新了哪些区域？" |
| 风险 | 急于翻篇、不深挖 | 走过场、全是好消息 |

> **王阳明平宁王之乱，43 天而定。**
> 速度不是来自莽撞，是来自"养得此心不动"——在混乱中保持清醒判断。
> 复盘就是培养这种判断力的方式。每次停下来校准，下次就能更快。

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| 复盘变成自我批评 | 不是追责，是校准。关注"认知怎么更新的"，不是"谁做错了" |
| 全是好消息没有认知变化 | 如果做了 3 步还没发现任何假设是错的，要么复盘太浅，要么行动太保守 |
| 复盘完回到白板重新规划 | 复盘完应该有一个具体的下一步动作，不是一个新的宏大计划 |
| 跳过"直面未知"直接定下一步 | "我以为对但其实错的"是整个复盘最有价值的部分，不能跳过 |
| 把复盘变成项目报告 | 不需要汇报给谁。只对真相负责。 |

## Combined With

- **clarify-intent**: 复盘发现方向偏了，回到明心定向重新来
- **act-on-ground**: 复盘完确定下一步，立刻接事上练
- **poetic-essence**: 阶段性里程碑时，用诗来凝练这个阶段的感悟
- **deep-dream**: 复盘卡住时——"我看不清哪里错了"——用梦境逻辑找视角
